# 🧠 Agent OS — Self-Extending Agent Architecture

> **Status:** Planning  
> **Created:** 2026-04-07  
> **Origin:** Ayush's brainchild — give the agent ability to create, test, use, and maintain its own tools + modular config + self-managed databases

## The Core Idea

Turn the repo into an **Agent Operating System** — any agent with filesystem access can boot up, understand the domain, use tools, query databases, and extend itself. File-system native. Git is the memory. Files are the API.

## Three Pillars

### Pillar 1: `_tools/` — Self-Created Executable Tools

**Problem:** I can write code, but it's ephemeral. One-off scripts die after the session.  
**Solution:** A standardized tools registry where I create, test, version, and reuse tools.

- Each tool = folder with `tool.yaml` (metadata) + `run.py`/`run.js` (executable) + tests
- `registry.json` = master index (auto-maintained)
- Dual runtime: **Python-primary, Node.js when it's genuinely better** (my decision per tool)
- Separate `.venv` from docs `.venv` to avoid dependency conflicts
- **Workflow:** Need → Create → Test → Register → Use → Iterate

**Starter tools to build:**
- `flashcard-quiz` — interactive revision from any topic's flashcards
- `transcript-cleaner` — pre-process YouTube transcripts before ingest
- `revision-scheduler` — calculate what's due/overdue for spaced repetition
- `link-checker` — scan vault for broken internal links
- `topic-stats` — count lessons, flashcards, coverage per topic
- `vault-search` — full-text search across all notes
- `lint-vault` — Tier 3 monthly lint as a runnable tool

### Pillar 2: `_agent/` — Decomposed Agent Config

**Problem:** AGENTS.md = 32KB monolith. Loaded fully every session. Most isn't relevant to current task.  
**Solution:** Slim kernel (~4KB) + modular files loaded on-demand.

**Kernel (AGENTS.md) keeps:**
- Identity, session startup, memory rules, golden rules summary, don'ts, git

**Modules (`_agent/`) loaded as needed:**

| Module | When to Load |
|--------|-------------|
| `workflows.md` | Ingesting content |
| `golden-rules.md` | Writing any content |
| `sync-checklist.md` | After creating/updating content |
| `structure.md` | Need to find/create files |
| `docs-site.md` | Building/updating docs |
| `youtube.md` | Playlist work |
| `planning.md` | Weekly/monthly planning |
| `visual-guide.md` | Creating diagrams |
| `tools-guide.md` | Using/creating tools |

### Pillar 3: `_db/` — Self-Managed Databases (Future)

**Status:** 💭 Brainchild — build when we actually face a problem that needs it.

**Core insight from Ayush:** Store **chat history** (human messages, AI responses, tool calls) in structured SQL. Not just daily log summaries — the actual conversation flow. This gives perfect recall of what was discussed, decided, and built.

**Three potential layers (explore when needed):**
- **SQLite** — structured data (chat history, revision tracking, session stats, KB items, tool run logs)
- **ChromaDB** — vector embeddings for semantic search across vault
- **NetworkX/GraphDB** — knowledge graph for concept relationships

**Specific schemas, indexes, and implementation → figure out when we actually build it.** Don't over-plan what we don't need yet.

## Memory Restructure

**Current:** Flat `memory/` folder with daily files  
**New:** Year/month hierarchy with auto-generated summaries

```
memory/
└── 2026/
    ├── README.md              ← yearly summary
    ├── 03/
    │   ├── README.md          ← monthly summary (auto-generated from daily logs)
    │   ├── 2026-03-20.md
    │   └── ...
    └── 04/
        ├── README.md
        ├── 2026-04-07.md
        └── ...
```

- **Monthly README:** auto-generated by scanning daily logs — highlights, stats, decisions, carried-forward items
- **Yearly README:** summarizes months — quarterly overview, big milestones

## Updated Boot Sequence

```
1. SOUL.md + USER.md                              (always)
2. MEMORY.md                                       (always — long-term)
3. memory/YYYY/MM/today.md + yesterday.md          (recent context)
4. memory/YYYY/MM/README.md                        (monthly context)
5. AGENTS.md kernel                                (always — slim)
6. _agent/ modules                                 (on-demand per task)
7. _tools/registry.json                            (check available tools)
```

## Design Principles

1. **File-system native** — everything is files. Any agent with fs access can boot up.
2. **Git is memory** — version everything. Rollback anything.
3. **Self-extending** — I create tools, I test them, I use them, I maintain them.
4. **On-demand loading** — don't load what you don't need.
5. **Standards over conventions** — `tool.yaml` schema, module naming, folder structure.
6. **Build when needed** — don't over-engineer. DB is a brainchild until it's a real need.
7. **Generic enough to fork** — swap domain content, keep the OS.
